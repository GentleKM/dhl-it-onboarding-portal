# Git 커밋 규칙

## 공통 규칙 (항상 적용)
- 커밋 전 `npm test` 실행 — 전체 통과 후 커밋 (실패 시 커밋 불가)
- 커밋 전 관련 docs/ 문서와 status.json을 최신화
- main 브랜치 강제 푸시(git push -f) 절대 금지
- 커밋 메시지·브랜치명은 모두 영문으로 작성
- 커밋 메시지 형식:
  - Phase 작업: `[Phase N] 작업 요약` (예: `[Phase 2] 입력 폼 구현`)
  - Phase 외 기능 추가: `[feat|fix] 기능/작업 요약` (예: `[feat] 다크모드 지원`, `[fix] 이메일 발송 오류 수정`)

## "배포 전" 과 "배포 후" 구분해서 규칙 적용
- 서비스가 실제 사용자에게 배포(production deploy) 되기 전에는 "배포 전" 규칙 적용
- 배포 후에는 사용자에게 전환 사실을 알린 후, "배포 후" 규칙 적용

### [배포 전] 서비스가 실제 사용자에게 배포되기 전까지 사용, 배포 후에는 비활성화
- 커밋 단위: Phase 1개 완료 = 커밋 1개
- **커밋 전 Codex 검토 완료** 후 main에 커밋 (검토 완료 후 다음 Phase 진행)
- 기본 작업 브랜치: main에서 순차 진행
- 다음의 경우에만 별도 Worktree/브랜치에서 작업 후 검증 완료 시 merge:
  - 대규모 리팩터링 / 실험적 시도 / 기존 기능 손상 위험이 있는 변경
- 작은 작업까지 브랜치로 나누지 않는다 (과도한 분기 금지)

### [배포 후] 서비스가 실제 사용자에게 배포된 후부터 사용, 그 전까지는 비활성화
- main = 라이브 서비스 브랜치. 직접 커밋·작업 금지 (보호 대상)
- 평소 모든 작업은 develop 브랜치에서 진행
- 큰 기능은 feature/기능명 브랜치에서 작업 → develop으로 merge
- develop → main 반영: **Codex(리뷰어) 최종 확인 후** Pull Request를 통해 병합
- 핫픽스(긴급 수정): `[hotfix] 긴급 수정 요약` / hotfix/이슈명 브랜치 → main + develop 양쪽 반영
- main 직접 푸시 금지

## Phase 커밋 절차 (역할별)

Phase 하나가 완료된 후 커밋까지의 단계별 담당자:

| 순서 | 담당 | 작업 | 비고 |
|------|------|------|------|
| 1 | **Claude** | Phase 코드 구현 완료 | |
| 2 | **Claude** | `npm test` 실행 → 전체 통과 확인 | 실패 시 수정 후 재실행 |
| 3 | **Claude** | `codex:codex-rescue` 에이전트 자동 스폰 → AGENTS.md 체크리스트 리뷰 | npm test 통과 직후 자동 실행 |
| 4 | **Codex** | PRD·보안·테스트·컨벤션 체크리스트 수행 | AGENTS.md 체크리스트 기준 |
| 5 | **Claude** | Codex 피드백 반영 후 `npm test` 재확인 | 수정이 없으면 생략 |
| 6 | **Claude** | `[Phase N] 영문 요약` 형식으로 커밋 | |
| 7 | **Claude** | `git push origin main` | |

### npm test vs Codex 검토 — 역할 차이

| 항목 | `npm test` | Codex 검토 |
|------|------------|------------|
| **담당** | Claude 자동 실행 | Claude가 자동 스폰 (npm test 통과 직후) |
| **검사 내용** | 코드 버그, 엣지케이스 (기술적 정확성) | PRD 준수, 보안, UX, 코딩 컨벤션 |
| **판단 방식** | 객관적 (pass/fail) | 주관적 품질 판단 포함 |
| **트리거** | Claude 직접 실행 | codex:codex-rescue 에이전트 자동 스폰 |

→ 둘 다 필요: `npm test`는 버그를 잡고, Codex 검토는 품질·적합성을 확인

### [CI 파이프라인 구축 후] CI 파이프라인 구축 후 활성화, 그 전까지는 비활성화
- develop → main 반영 조건에 자동 테스트(CI) 전체 통과 추가
- 빌드·린트·유닛 테스트 자동화 설정 (GitHub Actions 또는 Vercel CI 활용)
- GitHub 브랜치 보호 규칙으로 "CI 미통과 시 main 병합 금지" 강제
