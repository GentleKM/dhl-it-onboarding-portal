# 이슈 기록 관리

## 발동 조건 (언제 이 skill을 쓰는가)
다음 중 하나에 해당하는 critical 이슈를 **조치 완료한 직후** 이 skill을 실행한다:
- 기능 오작동 (의도대로 동작하지 않음)
- 보안 취약점 (데이터 노출, 인증 우회 등)
- 데이터 손실·손상
- 동일 원인으로 2회 이상 재발한 이슈

단순 오타·사소한 스타일 수정 등 경미한 이슈는 기록하지 않는다.

## 수행 절차
1. `/issues` 폴더가 없으면 생성한다.
2. 파일명 `YYYY-MM-DD-short-title.md` (영문 소문자, 하이픈)로 새 파일을 만든다.
   - 예: `2026-05-24-login-token-expired.md`
3. 아래 포맷으로 내용을 채운다.
4. 동일한 이슈가 재발하지 않도록 아래 "## 4. 재발 방지 및 AI Instruction (Prevention)" 내용을 rules/ 에 반영한다.
5. 작성 후 사용자에게 기록 위치와 반영한 규칙을 보고한다.

## 파일 포맷

```markdown
# [이슈 제목]

- 발생일: YYYY-MM-DD
- 심각도: Critical / High
- 상태: 해결 완료 / 조치 중

## 1. 현상 (Symptom)
- Expected: (기대한 동작)
- Actual: (실제 발생한 동작)
- 재현 방법: (어떤 상황에서 발생하는지 단계별로)

## 2. 근본 원인 및 배경 (Root Cause)
- 근본 원인: (왜 발생했는지 한 줄 요약)
- 배경 설명: (비개발자도 이해할 수 있게 전체 그림과 맥락을 쉽게)

## 3. 조치 결과 (Resolution)
- (어떻게 해결했는지 단계별로 쉽고 자세하게)
- 변경 파일/위치: (수정한 파일 경로)

## 4. 재발 방지 및 AI Instruction (Prevention)
- 오류 패턴 및 현상: (오류 발생 패턴 및 현상을 명확하게 정리)
- 근본 조치: (구조적으로 재발을 막는 방법)
- AI 적용 규칙: (하네스 관점의 설정·프롬프트, 행동 지시형으로)
- 반영 위치: (CLAUDE.md / rules/ / hooks/ 중 어디에 추가했는지)
```

## 5. Hook 생성 검토: 아래 조건에 걸리면 반드시 사용자에게 hook 생성 제안, 임의로 생성하지 말고 반드시 사용자 승인 후 생성

### 즉시 대상: 1회 발생만으로도 hook 후보 — 비가역적·외부 피해
- 데이터 손실/손상: 사용자 데이터 삭제·덮어쓰기, DB 스키마 파괴
- 보안 노출: 인증·인가 우회, 개인정보·API키·비밀번호 외부 노출
- 비가역 작업: 운영(production) 환경 배포/삭제, git force push, 실제 결제·과금

### 반복 대상: 누적 시 hook 후보
- 동일 원인의 이슈가 3회 이상 재발

### Hook 제안 시 포함할 내용
- 무엇을 차단하는가 (차단 대상 명령/동작)
- 어느 시점에 작동하는가 (예: 커밋 전, 명령 실행 전)
- 제안만 하고, 실제 생성·적용은 사용자 승인을 받는다.
